## 1. D1.3.4 Scope (Evaluate-Only Gate)

### 1.1 What D1.3.4 Includes (Evaluate-Only)

D1.3.4 is the evaluate-only slice:
- `POST /v1/proposals/{proposal_id}/evaluate` with `ActionProposal` request body
- `EvaluateProposalResponse { decision, reason, matched_rule_ids, warnings }` response parsing
- Proposal persistence (gateway writes proposal to DB as FK prerequisite for later execution)
- `PolicyEvaluated` provenance event emitted internally by gateway after success
- MCP does NOT emit provenance directly — gateway owns PolicyEvaluated emission

### 1.2 What D1.3.4 Excludes (Evaluate in Later Slices)

D1.3.4 is **evaluate-only**. The following are excluded and belong to later slices:

| Item | Slice | Reason |
|------|-------|--------|
| Capability minting | D1.4 | Requires evaluate success + later endpoint |
| Authorization | D1.4 (internal) | Creates execution_id |
| Prepare execution | D1.5 | Requires capability_id |
| Execute tool | D1.7 | Blocked by capability semantics |
| Rollback preparation | D1.5 | Requires execution_id |
| Approval/reject tools | D1.7 | Backend endpoint does not exist |
| Tool dispatch wiring | D1.7 | Blocked by semantics gap |
| Direct provenance emission | Forbidden | Gateway emits PolicyEvaluated internally |

### 1.3 What D1.3.4 Forbidden Items Remain

## 2. Endpoint Contract

### 2.1 Route

```
POST /v1/proposals/{proposal_id}/evaluate
```

**Path Parameters**:
- `proposal_id` (string): The proposal ID, generated by MCP/client before evaluate call

**Request Body** (JSON):
```json
ferrum_proto::ActionProposal
```

**Response** (JSON, 200 OK):
```json
{
  "decision": "Allow",  // One of: Allow, Deny, Quarantine, RequireApproval, AllowDraftOnly
  "reason": "...",
  "matched_rule_ids": ["rule-1", "rule-2"],
  "warnings": []
}
```

**`Decision` enum variants** (from `ferrum_proto::Decision`):
- `Allow` — proposal approved as-is
- `Deny` — proposal rejected
- `Quarantine` — proposal flagged for manual review
- `RequireApproval` — requires explicit approval before execution
- `AllowDraftOnly` — only draft-mode execution permitted

**Error Responses**:
- `400 Bad Request`: Invalid proposal or missing required fields
- `401 Unauthorized`: Missing or invalid bearer token
- `404 Not Found`: Intent referenced by proposal not found
- `500 Internal Server Error`: Gateway error

### 2.4 ID Origins

| ID | Origin | Generated By |
|----|--------|--------------|
| `intent_id` | From `POST /v1/intents/compile` response `envelope.intent_id` | Gateway |
| `proposal_id` | Generated by MCP/client before evaluate | MCP/client (UUID v4 via `ProposalId::new()`) |

**Note**: `proposal_id` must be stable and deterministic or properly random. The gateway does NOT generate proposal_id — it expects the MCP/client to supply it and persists the proposal under that ID. Per E1 oracle decision, `ProposalId::new()` (UUID v4 random) is accepted for D1.3.4; UUID v5 deterministic derivation may be considered in a later slice.

## 3. Preflight Blockers (E1–E2)

### E1 — proposal_id Generation Strategy

**Issue**: D1.3.4 requires a `proposal_id` to be generated before calling evaluate. Doc 75 §2.2 states MCP/client generates `proposal_id`.

**Current State**:
- Doc 79 P1/P2 resolved compile's `principal_id` mapping
- D1.3.3 compile flow exists in code (commit 1db8285)
- `generate_proposal_id()` helper in `stage2_types` returns `ProposalId` (not `String`) via `ProposalId::new()`

**Decision Required**:
1. Accept `ProposalId::new()` (UUID v4 random) for D1.3.4?
2. Defer UUID v5 deterministic generation to later?

**Recommendation**: Accept `ProposalId::new()` (UUID v4 random) for D1.3.4. This matches the existing `generate_proposal_id()` implementation. UUID v5 deterministic derivation (from intent_id + tool_name) can be considered in a later slice if lineage needs improve.

**Dependencies**: None (can be resolved independently)

---

### E2 — evaluate Response Decision Parsing

**Issue**: `EvaluateProposalResponse.decision` is `crate::Decision` enum with 5 variants. MCP must parse all variants correctly.

**Current State**:
- `Decision` enum defined in `ferrum_proto::common.rs`: `Allow`, `Deny`, `Quarantine`, `RequireApproval`, `AllowDraftOnly`
- No MCP-side handling of decision result yet
- D1.3.3 compile flow exists (commit 1db8285) but does not call evaluate

**Decision Required**:
1. How to handle each Decision variant in MCP tool response?
2. How to handle `warnings` field — surface or log?

**Recommendation** (all 5 variants):
| Variant | MCP Handling |
|---------|--------------|
| `Allow` | Return success with full `EvaluateProposalResponse` body |
| `Deny` | Return `McpToolError::forbidden()` with reason string |
| `Quarantine` | Return `McpToolError::quarantine()` with reason + matched_rule_ids |
| `RequireApproval` | Return `McpToolError::requires_approval()` with reason; surface warnings |
| `AllowDraftOnly` | Return `McpToolError::draft_only()` with reason; surface warnings |

Surface `warnings` in all responses for transparency. Deny/Quarantine/RequireApproval/DraftOnly carry `reason` for caller diagnostics.

**Dependencies**: None (can be resolved independently)
