## Why deferred

1. `sqlx::PgPool` already provides transparent reconnect with exponential backoff. A local Docker restart test (2026-05-21) confirmed ferrumd recovers without restart (14 s observed, target <= 30 s).
2. PG-2.3a (acquire-timeout metrics + pool-saturation readiness) gives fail-closed behavior when the pool is exhausted. The `/v1/readyz/deep` endpoint reports degraded when `idle_connections == 0 && total_connections >= max_connections`, and `ferrumgate_store_pg_acquire_timeouts_total` is emitted for observability.
3. A circuit breaker is only meaningful with a load-balancer or multi-node topology (PG-5 / HA). For the current single-node pilot, PgPool reconnect + `/readyz/deep` pool saturation + acquire timeout metric + alert templates + runbook are sufficient. Adding an app-level circuit breaker now would introduce complexity and false positives without a topology that can route around an unhealthy instance.

## B.3 decision — DEFER app-level PostgreSQL circuit breaker

**Verdict**: DEFER now. No circuit breaker ADR will be written until HA design begins.

**Rationale**:
- PgPool reconnect + `/readyz/deep` pool saturation + acquire timeout metric + alert templates + operator runbook are sufficient for single-node fail-closed behavior.
- A circuit breaker state machine adds value only when there is a load-balanced HA topology (PG-5) where per-instance health state and concrete RTO need definition.
- Premature implementation would add complexity, risk false positives, and require tuning thresholds without real multi-node failure data.

**Triggers for revisit** (any of):
1. PG-5 HA design begins (managed PG, Patroni, or load-balanced topology).
2. A load balancer is introduced in front of ferrumd or PostgreSQL.
3. RTO tightens beyond B.2's observed 14 s recovery and the operator requires faster/bounded failure detection.
4. Real-world PostgreSQL blips show failure modes that a half-open state could mitigate (e.g., intermittent connectivity that PgPool's exponential backoff does not handle well).

## B.4 decision — DEFER circuit breaker implementation

**Verdict**: DEFER until after HA ADR Phase 9 is approved and a multi-node/load-balanced topology is planned.

**Rationale**: Implementation depends on B.3 ADR. No code will be written before the ADR is approved.
