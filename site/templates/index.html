Intent-scoped execution gateway

FerrumGate is an audit and rollback layer for agent and operator actions. Before any side effect occurs, declare intent, get policy evaluation, obtain a time-bound capability, and leave a complete provenance trail.

Status: RC-ready / conditional. NOT production-ready. Suitable for local evaluation and single-node pilot only.

What problem FerrumGate solves

  • Unaudited automation — Agents and scripts mutate systems without a trace.
  • No rollback path — Mistakes are expensive because there is no compensation plan.
  • Policy bypass — Tools run directly against APIs/files/databases, skipping governance.
  • Approval gaps — High-risk actions proceed without human review.

FerrumGate inserts a gateway between intent and side effect: compile, evaluate, mint, prepare, execute, verify, and record. Every step is in the provenance graph. Every mutation has a rollback class.

When to use FerrumGate

  • You run automation (CI, agents, operators) that mutates files, git repos, databases, or HTTP APIs.
  • You need an audit trail of who/what initiated an action and whether it succeeded.
  • You want automatic compensation for reversible mistakes (R1).
  • You need a policy gate before critical actions (R3) with human approval.
  • You want a uniform MCP-compatible interface for diverse backends.

When NOT to use FerrumGate

  • High-throughput low-latency paths — The gateway adds policy, provenance, and verification overhead. It is not a zero-cost proxy.
  • Already-governed systems — If your platform already provides intent tracking, policy, and audit, adding FerrumGate may be redundant.
  • Simple read-only queries — R0 paths are supported, but a plain API client is simpler if you never mutate.
  • Production without operator readiness — FerrumGate requires backup, token rotation, monitoring, and incident response discipline. See the operator guide.

Architecture and lineage

FerrumGate uses a directed provenance graph to track every execution. The minimum lineage chain before a side effect is committed: 

Intent → Proposal → PolicyEvaluated → CapabilityMinted
  → SideEffectPrepared → ToolCallPrepared → ToolCallExecuted
  → SideEffectVerified → Terminal (Committed | Compensated | RolledBack)

Adapters implement prepare/execute/verify/compensate for specific domains: filesystem, git, HTTP, SQLite, and mail drafts. Each adapter declares a default rollback class per operation.

For a deeper explanation, see the Concepts guide.

Quickstart

The validated local demo path runs in under 30 minutes:

  1. Clone the repository and build with cargo build --release.
  2. Start ferrumd with the dev config: in-memory SQLite, loopback binding, auth disabled.
  3. Run the curl sequence from /v1/healthz through /v1/provenance/lineage.
  4. Try ferrumctl for CLI-driven operations.
  5. Configure an MCP client to interact via the MCP bridge.
